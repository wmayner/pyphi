# IIT 3.0 Tie Resolution

## Problem

The IIT 3.0 path has three tie-resolution issues that surfaced during
the IIT 3.0 restoration project and need to be settled for 2.0:

**(1) Cross-subsystem complex selection is incidental.** When two
candidate subsystems tie at maximum |big_phi| during major-complex
selection, PyPhi falls through to `partition.lex_key()` on the
within-subsystem MIP. This is a property of one specific cut inside
one subsystem being used to choose between two different subsystems â€”
reproducible but architecturally wrong.

The worked example: substrate `standard` at `current_state=(0, 0, 1)`
has two SIAs that tie at |big_phi| = 1.0 â€” the subsystem over nodes
`(1, 2)` (B=AND, C=XOR) and `(0, 2)` (A=OR, C=XOR). These are
physically distinct subsystems with different gate compositions, not
symmetry equivalents. PyPhi 1.x picked `(1, 2)` by iteration order;
the post-`82b778ca` route picks `(0, 2)`. Albantakis et al. 2019
(p. 17) explicitly notes that symmetric ties of this form are
"undetermined". For the transition `(1, 0, 0) â†’ (0, 0, 1) â†’ (1, 1, 0)`,
`true_events` returns 2 events under `(1, 2)` and 0 events under
`(0, 2)`. `test/test_actual.py::TestActualCausationIIT30::test_true_events`
is `@pytest.mark.skip` pending policy.

**(2) `sia_tie_resolution` is dead code on the IIT 3.0 path.** IIT 4.0's
`_find_mip_for_fixed_state` runs `MapReduce` â†’ `resolve_ties.sias()`,
which consults `config.formalism.iit.sia_tie_resolution`. IIT 3.0's
`_sia_map_reduce` uses `MapReduce(reduce_func=min)` directly via
`OrderableByPhi.__lt__`, never consulting the config knob. Surface
asymmetry between the formalisms; the comment in `presets.iit3`
claiming the knob is "unused on the IIT 3.0 code path" treats the
asymmetry as intentional, but it isn't.

**(3) `mip_tie_resolution` default applies IIT 4.0 normalization on
the IIT 3.0 path.** `formalism.queries.find_mip` â†’ `_find_mip_single_state`
â†’ `resolve_ties.partitions(candidate_mips)` is shared by both
formalisms. The default `["NORMALIZED_PHI", "NEGATIVE_PHI"]` selects
`argmin normalized_phi` (then argmax phi) where
`normalized_phi = phi / num_connections_cut(partition)` per the
`NUM_CONNECTIONS_CUT` registry. The IIT 3.0 paper (Oizumi et al. 2014)
defines the MIP as raw `argmin Ď†(cut)` with no normalization â€” 3.0
dropped the system-level normalization that Tononi et al. 2008 used,
and never introduced a mechanism-level normalization. The current
default is non-paper-canonical for 3.0 and can in principle select a
different candidate than `argmin(raw Ď†)` when candidates have
coincident normalized_phi but different raw phi.

## Goals

1. Replace the incidental cross-subsystem tie-break with a
   Determinism-level cascade that signals indeterminacy when the
   tie cannot be resolved, mirroring IIT 4.0's exclusion-failure
   handling.
2. Route IIT 3.0's within-subsystem system-MIP selection through
   `resolve_ties.sias()` with a 3.0-specific
   `sia_tie_resolution = ["PHI", "PARTITION_LEX"]` default, unifying
   the architecture across formalisms.
3. Set IIT 3.0's `mip_tie_resolution = ["PHI", "PARTITION_LEX"]`
   default in `presets.iit3`, restoring paper-faithful raw-Ď† MIP
   selection.
4. Document the four IIT 3.0 tie-resolution paths and which knobs
   apply in the `presets.iit3` comment block.
5. Unskip `test_true_events` with updated expectations matching the
   chosen policy.

## Tie-resolution paths on the IIT 3.0 path

Four paths, three of which this spec changes:

| Path | Where | Mechanism | Config knob | Status after this spec |
|---|---|---|---|---|
| Mechanism MIP | `formalism/queries.py:_find_mip_single_state` | `resolve_ties.partitions` | `mip_tie_resolution` | **Preset changed** to `["PHI", "PARTITION_LEX"]` |
| Purview at MICE | `formalism/queries.py:289` | `resolve_ties.purviews` | `purview_tie_resolution` | Already `["PHI", "PURVIEW_SIZE"]` in preset; unchanged |
| System MIP within subsystem | `formalism/iit3/__init__.py:_sia_map_reduce` | Refactored to `resolve_ties.sias` | `sia_tie_resolution` | **Newly applies**: preset set to `["PHI", "PARTITION_LEX"]` |
| Cross-subsystem complex | `substrate.complexes` | New `_iit3_exclusion_cascade` | none (hard-coded cascade) | **New behavior**: indeterminate on tie |

## Policies

### Cross-subsystem complex selection

When two subsystems tie at maximum |big_phi|, the cascade returns
`UNRESOLVED_WITHIN_BUDGET` carrying the tied set and a lex-smallest
diagnostic representative. The caller in `_iit3_exclusion_cascade`
interprets `UNRESOLVED_WITHIN_BUDGET` as "no complex from this
clique" â€” same semantic as IIT 4.0's Composition-escalation failure
in `_resolve_clique_by_big_phi`. Those tied candidates are skipped;
the tier walk proceeds to the next overlap-clique or next tier. If
no clique resolves at any tier, `maximal_complex()` returns a null
SIA over the empty system.

For the worked example, `(1, 2)` and `(0, 2)` overlap on node 2,
form one clique, are indeterminate, and the clique falls out.
`maximal_complex(standard, (0, 0, 1))` returns the null SIA;
`true_events` returns 0 events. The IIT 3.0 paper's "undetermined"
language is reflected directly: the formalism doesn't pick, so PyPhi
doesn't pick.

This is the most conservative reading of the paper. PyPhi 1.x picked
`(1, 2)` by iteration order, but that was an arbitrary
implementation artifact, not principled.

### System-MIP selection within a subsystem

Refactor `pyphi/formalism/iit3/__init__.py:_sia_map_reduce` to call
`resolve_ties.sias()` on the candidate set produced by `MapReduce`,
replacing the in-band `reduce_func=min`. `presets.iit3` adds:

```python
sia_tie_resolution=["PHI", "PARTITION_LEX"],
```

The `PHI` strategy (with `resolve_ties.sias`'s `operation=min`)
selects `argmin Ď†` â€” the MIP by Oizumi 2014's raw-Ď† definition.
`PARTITION_LEX` breaks ties deterministically to the lex-smallest
partition. Observable behavior is identical to the current
`OrderableByPhi.__lt__` route via `MapReduce(reduce_func=min)`,
because `IIT3SystemIrreducibilityAnalysis.order_by = (phi,
partition.lex_key())` already keys on `(Ď†, partition.lex_key())`.

### Mechanism-MIP selection

`presets.iit3` adds:

```python
mip_tie_resolution=["PHI", "PARTITION_LEX"],
```

The current shared default `["NORMALIZED_PHI", "NEGATIVE_PHI"]`
filters to `argmin normalized_phi` then `argmax raw Ď†`. IIT 3.0
defines the MIP as `argmin raw Ď†` with no normalization. The new
default uses `PHI` (raw Ď†) as the primary key and `PARTITION_LEX`
as the deterministic tiebreak.

This is **not observably a no-op**. When two partitions have the
same `normalized_phi` but different raw Ď†, the current default
prefers the larger-Ď† partition; the new default prefers the
smaller-Ď† partition. Goldens may shift. Concrete example: candidate
A with Ď†=1.0 and num_connections_cut=2 has normalized_phi=0.5;
candidate B with Ď†=0.5 and num_connections_cut=1 also has
normalized_phi=0.5. Current default picks A; the new default
picks B (which is the actual raw-Ď† MIP).

## Non-goals

- Refactoring the four IIT 3.0 paths into a single dispatch
  abstraction. Each remains explicit; the spec only changes the
  per-path tie-resolution behavior.
- Introducing a new IIT 3.0-specific normalization. The 2014 paper
  has none and PyPhi adds none.
- Changing IIT 4.0 behavior. Defaults stay; only the IIT 3.0
  preset gets new values for `mip_tie_resolution` and
  `sia_tie_resolution`.

## Architecture

### Cross-subsystem cascade resolver

Add to `pyphi/resolve_ties.py`:

```python
class _IIT3SiaLike(Protocol):
    """Structural type for an IIT 3.0 SIA consumed by the
    cross-subsystem cascade."""

    @property
    def node_indices(self) -> Sequence[int]: ...


def resolve_iit3_complex_tie[V: _IIT3SiaLike](
    sias: "Iterable[V]",
    *,
    context: ResolutionContext,
    on_unresolved: OnUnresolved = "defer",  # noqa: ARG001
) -> CascadeOutcome[V]:
    """Resolve a cross-subsystem complex tie on the IIT 3.0 path.

    The 2019 AC paper (p. 17) notes that symmetric ties of this
    form are undetermined; the 2014 IIT 3.0 paper provides no
    additional postulate to escalate to. The cascade returns the
    lex-smallest candidate as a diagnostic representative but
    flags the outcome ``UNRESOLVED_WITHIN_BUDGET`` so callers
    treat the clique as failing the exclusion postulate (no
    complex).
    """
    survivors = tuple(sias)
    if not survivors:
        raise ValueError("resolve_iit3_complex_tie requires at least one SIA")
    representative = min(survivors, key=lambda s: tuple(sorted(s.node_indices)))
    return CascadeOutcome(
        resolved=representative,
        tied_set=survivors,
        cascade_level="Exclusion",
        outcome="UNRESOLVED_WITHIN_BUDGET",
    )
```

The structure mirrors `resolve_ac_causal_link_tie` (line 447-490)
exactly: a representative for diagnostic display, the full tied
set for inspection, an `UNRESOLVED_WITHIN_BUDGET` flag for the
caller to decide what to do.

### Substrate-level wiring

Add to `pyphi/substrate.py`:

```python
def _resolve_clique_iit3(clique: list[Any]) -> Any | None:
    """Return the unique complex of an IIT 3.0 overlap clique, or
    ``None`` when the clique is indeterminate.

    Single-candidate cliques resolve immediately; multi-candidate
    cliques run through ``resolve_iit3_complex_tie``, which always
    flags ``UNRESOLVED_WITHIN_BUDGET`` (no paper-canonical
    escalation exists). The caller skips indeterminate cliques.
    """
    from pyphi import resolve_ties

    if len(clique) == 1:
        return clique[0]
    ctx = resolve_ties.ResolutionContext(max_escalation_level="Exclusion")
    outcome = resolve_ties.resolve_iit3_complex_tie(clique, context=ctx)
    if outcome.outcome == "RESOLVED" and outcome.resolved is not None:
        return outcome.resolved
    return None


def _iit3_exclusion_cascade(
    sorted_sias: list[Any],
    substrate: Substrate,                       # noqa: ARG001 â€” kept for parity
    state: tuple[int, ...],                     # noqa: ARG001 â€” kept for parity
) -> list[Any]:
    """Walk SIAs in descending |big_phi| tiers, applying the IIT 3.0
    cross-subsystem cascade within each overlap clique.
    """
    result: list[Any] = []
    covered: set[int] = set()
    for tier in _phi_groups(sorted_sias):
        survivors = [
            sia for sia in tier
            if not (set(_sia_node_indices(sia) or ()) & covered)
        ]
        if not survivors:
            continue
        for clique in _find_overlap_cliques(survivors):
            winner = _resolve_clique_iit3(clique)
            if winner is not None:
                _accept(winner, result, covered)
    return result
```

`complexes()` dispatches on IIT version:

```python
if _config_iit_version() == "IIT_3_0":
    return _iit3_exclusion_cascade(sorted_sias, substrate, state)
return _substrate_exclusion_cascade(sorted_sias, substrate, state)
```

The previous `_greedy_condensation` is no longer reachable and is
removed.

### `_sia_map_reduce` refactor

Replace the in-band `reduce_func=min` in
`pyphi/formalism/iit3/__init__.py:_sia_map_reduce` with a list
collection + `resolve_ties.sias()` filter:

```python
def _sia_map_reduce(
    cuts: Iterable[DirectedBipartition],
    system: System,
    unpartitioned_ces: Distinctions,
    **kwargs: Any,
) -> IIT3SystemIrreducibilityAnalysis:
    from pyphi import resolve_ties

    kwargs = {**dict(config.infrastructure.parallel_partition_evaluation), **kwargs}
    candidates = MapReduce(
        evaluate_partition,
        cuts,
        map_kwargs={
            "unpartitioned_system": system,
            "unpartitioned_ces": unpartitioned_ces,
        },
        shortcircuit_func=utils.is_falsy,
        desc="Evaluating cuts",
        **kwargs,
    ).run()
    null = _null_sia(system)
    if not candidates:
        return null
    ties = tuple(resolve_ties.sias(candidates, default=null))
    return ties[0] if ties else null
```

This route consults `config.formalism.iit.sia_tie_resolution`,
which the new preset sets to `["PHI", "PARTITION_LEX"]`.

## Risk and rollback

- **Risk**: items 1-2 (cross-subsystem cascade + `_sia_map_reduce`
  refactor) inadvertently change a golden outside the worked
  example. *Mitigation*: the implementation order runs goldens
  between item 4 and item 7 so any unexpected change is caught
  before the mechanism-MIP correction lands.
- **Risk**: item 3 (`mip_tie_resolution` change) shifts more
  goldens than expected. *Mitigation*: the audit step explicitly
  diffs file hashes; each shift is verified to reflect raw-Ď† vs
  normalized-Ď† MIP selection on a tied candidate, not a different
  bug.
- **Risk**: the lex-smallest representative leaks into a
  user-visible API and is misinterpreted as a canonical choice.
  *Mitigation*: representative is only used for diagnostic display
  in the `CascadeOutcome.resolved` field; `_resolve_clique_iit3`
  discards it when the outcome is `UNRESOLVED_WITHIN_BUDGET`. No
  public API surfaces it.
- **Rollback**: the three changes are separable commits. Reverting
  item 3 alone restores the previous mechanism-MIP behavior;
  reverting items 1-2 restores the previous cross-subsystem and
  within-subsystem behavior. Goldens regenerated for item 3 stay
  paper-canonical regardless of revert direction.

## Audit-process post-mortem

The `mip_tie_resolution` issue (item 3 of the goals) is a real
correctness bug that was anticipated, classified as "probably no
effect", and not confirmed before goldens were locked. Worth
documenting why.

### Sequence

- **2026-05-12 20:43 (`760bf3bf`)**: IIT 3.0 restoration regenerates
  four EMD goldens. Cross-checks `basic_iit3_emd` SIA-phi against
  `test/data/iit3-canonical/basic_sia_phi_canonical.json` â€” passes,
  because the bug doesn't propagate to SIA-phi on the default BI
  scheme.
- **2026-05-12 20:55 (`ee4391aa`)**: Restoration audit close-out.
  Catches `purview_tie_resolution` via direct measurement on
  `rule110 (1,0,1)` and `grid3 (1,0,0)` substrates. Reconciles to
  PyPhi 1.x's value. Catalogues `mip_tie_resolution` and
  `sia_tie_resolution` as "deferred to P11.95d" with the reasoning
  "IIT 3.0 has a single MIP per (mechanism, purview) by
  construction, but worth confirming".
- **2026-05-17 (this spec, first draft)**: Brainstorm of P11.95d
  enumerates the four tie-resolution paths; spec covers
  cross-subsystem cascade only; `mip_tie_resolution` change
  flagged as "post-2.0 non-goal" on the (still-untested) assumption
  that the 4.0 normalization on the 3.0 path is "deterministic but
  not paper-canonical", with no measurement.
- **2026-05-18 (this spec, second draft)**: Mid-discussion, a
  5-minute confirmation experiment under
  `mip_tie_resolution=["PHI", "PARTITION_LEX"]` shows three of four
  IIT 3.0 goldens shift, including `basic_iit3_emd_tri`'s SIA-phi
  (2.520833 paper-canonical vs 2.8125 in the locked golden). The
  deferred experiment is the experiment that should have run six
  days earlier.

### Why it wasn't caught

1. The audit deferred a 5-minute confirmation to a future
   brainstorm based on a structural assumption ("3.0 has a single
   MIP per (mechanism, purview) by construction") that was never
   tested. The assumption is false: 3.0 can have multiple
   bipartitions tied at minimum Ď†, and `resolve_ties.partitions`
   then breaks the tie using the buggy default.
2. The golden cross-check validated `basic_iit3_emd` SIA-phi
   against a canonical reference. SIA-phi happens to be unchanged
   under the fix for the BI-scheme fixtures â€” the mechanism-MIP
   shifts don't propagate to SIA-phi for `basic`, `xor`, or
   `basic_subset`. The one fixture where the bug shifts SIA-phi
   (`basic_iit3_emd_tri`, WEDGE_TRIPARTITION scheme) has no
   canonical reference; its value was locked to whatever the code
   produced.
3. The `purview_tie_resolution` issue was caught because someone
   ran `sia.phi` measurements on tie-prone substrates. The
   analogous measurement for `mip_tie_resolution` was not run.

### Lessons

- **Confirmation experiments are part of the audit, not future
  work.** A 5-minute experiment that would have falsified the
  deferral was deferrable in name only; running it would have
  collapsed the post-2.0 brainstorm into a same-session fix and
  prevented six days of goldens encoding buggy values. The cost
  of confirmation is fixed; deferral only multiplies the
  revalidation cost when the assumption is wrong.
- **Golden cross-checks at SIA-phi only are insufficient.**
  Mechanism-level partition shapes can shift without SIA-phi
  changing, hiding the disturbance. Coverage needs to extend to
  mechanism-MIP shapes or per-mechanism small-Ď† tuples.
- **Non-default schemes (TRI, etc.) need their own canonical
  references.** The TRI fixture is the only one with an SIA-level
  shift, and it had no independent reference because the
  `basic_sia_phi_canonical.json` file only covers the default BI
  scheme.
